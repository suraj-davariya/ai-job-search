## 0. Why this proposal exists

The DD found that our **breakout** competitor (Career-Ops, ~54K★) has independently converged on most of our planned feature set — reviewer, interview prep, cover letters, a tracker, pluggable portals, even a LaTeX path. Re-reconciled against our *actual* repo, our spec is mature and already covers those. So this proposal is **not** a rebuild. It is:

1. **Global reach (Pillar 1)** — the owner's headline directive: make the whole product speak every language. This is now our **primary strategic wedge**, not a side feature — and our prompt-as-code nature gives us a structural head start no SaaS competitor has.
2. **The genuine gaps the DD exposed** (Pillars 2–7) — trust/safety scam-shield, a *visible* no-fabrication guarantee, a guaranteed-ATS export, a token-free scan tier, liveness re-checks, and provider resilience.

Each pillar lists: the insight, what exists today, the proposed additions (with draft IDs), priority (MoSCoW), and where it lands in `/docs`.

## Pillar 1 — Global Internationalization & Localization (HEADLINE)

**Insight / directive.** "Each language we add is real value to that language-speaking community." Every surface a user reads, interacts with, or converts (README, docs-site, command output, dashboard UI, glossary, error messages, generated documents) should be available in their language, and adapt to their region's conventions.

**Structural advantage.** Because user-facing *content* (CV bullets, cover letters, evaluations, interview answers) is generated by the model at runtime, CareerForge is **already multilingual at the content layer** — a head start SaaS tools with hardcoded English UIs don't have. What's missing is (a) translating the **static chrome** and (b) **region/locale** adaptation.

**Two axes, kept deliberately separate** (this distinction is the architectural core):
- **Language** = the words the UI/docs are written in (translate static strings).
- **Locale / region** = conventions for that market (CV-vs-résumé, photo norms, page count, date/currency/name order, GDPR & anti-discrimination clauses). Pluggable **locale packs**, extensible like portal adapters.

**What exists today.** `NFR-0006` (cover-letter language matching + date/closing phrases); a hardcoded "**CV always English**" rule (`05-cv-templates.md`); English-only README, docs-site, dashboard UI, and CLI output.

**Proposed additions** → new `docs/architecture/adr-0007-internationalization-localization.md` (Proposed) + new `docs/requirements/functional-requirements-internationalization.md` (REQ-7xxx):

| ID | Requirement | Priority |
|----|-------------|----------|
| REQ-7001 | **Language-agnostic runtime output** — generate all user-facing content in the user's chosen language; replace the hardcoded "CV always English" with a locale-pack-driven decision (CV language = posting language OR user preference). | Must |
| REQ-7002 | **Externalized static strings** — all CLI/command output, prompts, and dashboard UI chrome moved out of code into translation resources (English = source of truth). | Must |
| REQ-7003 | **Translation resource format** — ICU MessageFormat in JSON, BCP-47 dirs `i18n/<lang>/…`, named placeholders (`{name}`), and CLDR plural categories (`zero/one/two/few/many/other`). ICU chosen over gettext PO because Arabic, Slavic (ru/uk/pl), and Indic languages have 3–6 plural forms PO can't express correctly. | Must |
| REQ-7004 | **Language roadmap & tiers** — the published Tier-1/Tier-2 lists in §1a + a path for *any* community to add its language. | Must |
| REQ-7005 | **Translation contribution workflow** — a translator CONTRIBUTING guide; **machine-translation seed + human post-edit (MTPE)** as the default seeding model; PR-reviewed like code; contributors credited via git history + the platform's per-translator stats. | Should |
| REQ-7006 | **Translation parity / staleness gate** — **custom CI check** (no off-the-shelf tool does this): each translated file records the **source commit-hash / release-tag** it was based on in front-matter; CI diffs the current English hash against it and surfaces "N strings / commits behind." Mirrors the spirit of our existing `docs-site/` parity test. | Must |
| REQ-7007 | **README & docs-site translation** — localized README quick-starts + a docs-site language switcher; user-facing docs fully translatable. | Should |
| REQ-7008 | **Dashboard UI localization + RTL** — dashboard renders in the chosen language with full right-to-left support (`ar`, `he`, `fa`, `ur`): bidi-aware layout, mirrored UI, logical CSS properties, plus CJK line-breaking/full-width punctuation and Indic-shaping-capable fonts. | Should |
| REQ-7009 | **Locale packs (pluggable region conventions)** — a documented locale-pack interface, **decoupled from UI language**: document term (CV/résumé/Lebenslauf/rirekisho/currículo), page-count, photo norm (forbidden/optional/expected/required-with-spec), included personal fields (DOB, marital status, nationality, national ID), name ordering, date/number/currency (CLDR), address format, privacy/legal clauses. Community-extensible. Directly realizes CLAUDE.md's "no hardcoded locales." | Must |
| REQ-7010 | **Locale-aware document generation** — CV & cover-letter templates consume the active locale pack (e.g. no photo/DOB in US/UK; photo + DOB expected in DE; Japan's two-doc rirekisho/shokumukeirekisho; CPF + marital status in BR; A4 vs Letter; GDPR consent line in EU). | Should |
| REQ-7011 | **Per-language generation-quality signal + HITL for low-resource languages** — LLM translation/generation quality degrades materially for low-resource languages (measured gaps up to ~24%; risk of mistranslation, hallucination, or silently switching to a related language). For lower-resource Tier-1/2 locales (e.g. `bn`, `ta`, `te`, `sw`, `ur`), surface a quality caveat and lean on existing human-in-the-loop review (ARCH-0006). Do **not** assume uniform output quality across the tier list. | Should |

**Proposed NFRs:** NFR-0019 RTL & non-Latin/Indic/CJK script rendering (UI **and** LaTeX pipeline — `lualatex`/`xelatex` need shaping-capable fonts bundled); NFR-0020 translation-completeness threshold to mark a language "released" vs "beta."

**Scoping decision to confirm with you (Open Question #1).** Recommendation, and it matches industry precedent (React, Vue, MDN, TypeScript, Kubernetes, Docusaurus all do exactly this): keep the **deep internal engineering spec** (`docs/requirements`, `docs/architecture` — 130+ REQ/ARCH/TC docs, contributor-facing, fast-churning) **English-canonical**, while **fully translating every user-facing surface** (README/quick-start, docs-site guides, command output, dashboard, glossary, error messages). Translating 130+ internal specs into every language perpetually lags and pays little; translating what users actually touch is where each language genuinely serves its community. *You said "each docs" — this is the one place I'd push back. Your call (Open Question #1).*

### §1a. Language roadmap (data-grounded)

Tiering blends **internet-user reach × labor-market size/openness × cross-border mobility × growth** (sources: Ethnologue 2025 speaker counts; internet-users-by-language; web-content share; growth signals). Data is sourced; the **cut lines are an editorial proposal to validate.**

**Tier 1 — launch (~13):**
`en` English · `es` Spanish · `zh-Hans` Chinese (Simplified) · `hi` Hindi · `ar` Arabic 🔁RTL · `pt-BR` Portuguese (Brazil) · `fr` French · `de` German · `ja` Japanese · `ru` Russian · `id` Indonesian · `bn` Bengali · (`pt-PT` deferred to Tier 2)

> Rationale: top internet-user languages + three high-mobility/high-growth labor markets (Hindi, Bengali, Indonesian) that web-content-share data under-counts. German/French/Japanese earn their slot on labor-market size and web maturity despite smaller populations.

**Tier 2 — community-driven (~18):**
`ur` Urdu 🔁 · `fa` Persian 🔁 · `he` Hebrew 🔁 · `tr` Turkish · `vi` Vietnamese · `ko` Korean · `it` Italian · `pl` Polish · `nl` Dutch · `uk` Ukrainian · `th` Thai · `ms` Malay · `ta` Tamil · `te` Telugu · `mr` Marathi · `fil` Filipino · `sw` Swahili · `ro` Romanian · `zh-Hant` Chinese (Traditional)

**Script/directionality engineering flags (each gated as its own work item):**
- 🔁 **RTL** (`ar`, `he`, `fa`, `ur`) — biggest layout item: bidi, mirrored UI, RTL-safe LaTeX. → NFR-0019.
- **CJK** (`zh-Hans`/`zh-Hant`, `ja`, `ko`) — no word-spacing, line-break rules, full-width punctuation, glyph bundling; **Simplified ≠ Traditional** (separate locales).
- **Indic shaping** (`hi`, `bn`, `ta`, `te`, `mr`) — needs shaping-capable fonts in **both** UI and the LaTeX pipeline.
- **Cyrillic** (`ru`, `uk`) — font coverage only, low risk.

**Tooling recommendation (Open Question #2):** **Weblate** as default — GPL-3, fully FOSS, self-hostable, native Git/CI round-trip (no vendor lock-in, fits our open ethos); **Crowdin free-OSS tier** as the lower-effort hosted alternative. Both support ICU + MT pre-translation. *"All languages in the world" is the north star;* the tier list is just the seeded order — the contribution workflow (REQ-7005) lets any community add and own its language.

## Pillar 2 — Trust & Safety: scam / ghost-job / posting-legitimacy shield

**Insight.** Career-Ops scores a dedicated **red-flags** dimension and emits a **legitimacy verdict** (High Confidence / Caution / Suspicious). Scam and ghost postings are rampant globally — *especially* in the international markets Pillar 1 opens us to. Protecting users here is both a safety duty and a differentiator.

**What exists today.** Five-dimension fit rubric (`REQ-2010`: Technical/Experience/Behavioral/Career/Location) with **no legitimacy or red-flags axis**; quick-fit is High/Med/Low only.

**Proposed additions** → new `docs/requirements/functional-requirements-trust-and-safety.md` (REQ-8xxx) + amend `business-rules-and-validation.md`:

| ID | Requirement | Priority |
|----|-------------|----------|
| REQ-8001 | **Posting legitimacy assessment** in `/apply` — a verdict (Verified / Caution / Suspicious) with cited reasons, surfaced before drafting. | Must |
| REQ-8002 | **Red-flags signal** added to evaluation — scam/ghost/exploitative-terms detection as a gate/flag (not silently folded into the 0–100 score). | Must |
| REQ-8003 | **Quick legitimacy flag** in `/search` results — cheap heuristic flag alongside the High/Med/Low fit signal. | Should |
| REQ-8004 | **Ghost-job & reposting signals** — detect long-open/re-posted/never-closing listings (ties to Pillar 6 liveness). | Could |
| REQ-8005 | **Scam-pattern catalog** — a community-extensible, **locale-aware** catalog (upfront fees, personal-data/ID harvesting, off-platform redirects, too-good comp, etc.). Patterns differ by region → ties to Pillar 1 locale packs. | Should |

**Glossary additions:** Posting Legitimacy, Red Flag, Ghost Job.

## Pillar 3 — Demonstrable No-Fabrication (turn our rule into a visible promise)

**Insight.** The DD's single strongest finding: Career-Ops's own advocates admit it **hallucinates skills onto CVs**, with no independent check. We have the *opposite* — `ARCH-0007`, the interview-backtrack test, an isolated reviewer, a verification checklist — but it's **invisible to the user**. Make the guarantee *demonstrable*: this becomes our headline differentiator.

**What exists today.** `ARCH-0007` + `REQ-2023` no-fabrication; reviewer reads 4 profile files; verification checklist runs once in `/apply`. No user-facing provenance artifact.

**Proposed additions** → amend `functional-requirements-application.md`:

| ID | Requirement | Priority |
|----|-------------|----------|
| REQ-2065 | **Fabrication audit artifact** — the verification step emits a claim → backing-profile-file ledger; any claim not traceable to the profile is flagged for the user, not silently shipped. | Must |
| REQ-2066 | **Surface provenance to the user** — show the audit in `/apply` output and as a dashboard "Provenance" panel per application ("every line on your CV, and where it came from"). | Should |

**Positioning:** README + docs-site lead with *"We never invent a skill you don't have — and an independent agent verifies every draft, with receipts."*

## Pillar 4 — Guaranteed-ATS export + parse self-check

**Insight (corrected after reading our code).** We are **not** ATS-blind — `cfcv.cls` is already single-column, no letter-spacing, standard section names, plain-text skills. The residual gap: an LLM-rendered LaTeX **PDF** can still trip some ATS parsers, and many ATS prefer plain text / `.docx`. So the add is a *guaranteed-parseable export* + a *verification that parsing actually works* — not a new ATS strategy.

**What exists today.** ATS-aware LaTeX class → one polished PDF.

**Proposed additions** → amend `functional-requirements-application.md` + note on `ADR-0003`:

| ID | Requirement | Priority |
|----|-------------|----------|
| REQ-2063 | **ATS-safe export variant** — emit a guaranteed-machine-parseable artifact (plain-text and/or `.docx`) from the same content model, alongside the polished PDF. | Should |
| REQ-2064 | **ATS parse self-check** — extract text back out of the generated PDF and assert key fields/keywords (name, contact, section headers, top skills) are recoverable; warn the user if not. | Should |

## Pillar 5 — Token-free deterministic "scan" tier

**Insight.** Career-Ops scans ATS platforms with **deterministic scrapers (no tokens)** before any LLM work. Our search is web-search/LLM-driven from the first step. A cheap deterministic pre-filter lowers cost and lets us scale coverage.

**What exists today.** `ADR-0004` pluggable-portal pattern (0 adapters built); `/search` is web-search-based; pre-fetch snippet filtering (`REQ-1004`) already trims tokens.

**Proposed additions** → amend `ADR-0004` + `functional-requirements-job-search.md`:

| ID | Requirement | Priority |
|----|-------------|----------|
| REQ-1013 | **Adapter `list()` contract** — portal adapters may expose a deterministic listing pass (title/company/url/date) that feeds search *before* any LLM fit assessment. | Could |
| NFR-0021 | **Cost-aware search** — deterministic pre-filtering is preferred over LLM calls wherever an adapter can provide structured listings. | Could |

## Pillar 6 — Active liveness re-check

**Insight.** Career-Ops Issue #373: months-closed jobs leaked into its pipeline because dedup ≠ liveness. We filter by *date* (`REQ-1011`) but don't re-verify a posting is still **open** at apply-time.

**Proposed addition** → amend `functional-requirements-job-search.md`:

| ID | Requirement | Priority |
|----|-------------|----------|
| REQ-1015 | **Liveness verification** — before `/apply` drafts for a stored posting, re-fetch/HEAD the URL and warn if it appears closed/expired. | Should |

## Pillar 7 — Provider resilience / graceful degradation

**Insight.** The DD surfaced a real Claude-native risk: a user reported Anthropic account bans during automated use. Our Claude-native stance is deliberate (depth > portability), but we should **document and degrade gracefully**, and not encourage abuse-tripping high-volume headless runs.

**Proposed addition** → amend `non-functional-requirements.md` (under `ARCH-0005`):

| ID | Requirement | Priority |
|----|-------------|----------|
| NFR-0022 | **Provider-limit resilience** — detect rate-limit/availability failures, degrade gracefully with a clear user message, and document safe-usage guidance (no mass headless batching). | Should |
