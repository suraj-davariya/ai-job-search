# Functional Requirements — Internationalization & Localization

> **Purpose:** Specifies the requirements for making CareerForge usable, readable, and convertible in many world languages, and adaptable to regional conventions — so each added language delivers real value to that language community.
>
> **Status:** Draft
> **Last updated:** 2026-06-18
> **Owner persona:** Localization Architect / Business Analyst

---

## Overview

CareerForge serves job seekers anywhere in the world (CLAUDE.md prime directive). This document specifies two **separate** capabilities:

- **Language** — translating every user-facing static surface (README/quick-start, docs-site, CLI/command output, dashboard UI, glossary, error messages). Internal engineering specs stay English-canonical.
- **Locale / region** — adapting document generation to a market's conventions (CV norms, photo rules, page count, date/currency, privacy/legal clauses), via pluggable **locale packs** decoupled from UI language.

Architecture and file organization are defined in [ADR-0007](../architecture/adr-0007-internationalization-localization.md). Runtime-generated content (CVs, cover letters, evaluations, interview prep) is already multilingual via the model; these requirements cover the static chrome and the locale layer.

### REQ-7001: Language-Agnostic Runtime Output
**Priority:** Must
**Description:** The system shall produce user-facing generated content in the user's chosen language, replacing the hardcoded "CV always English" rule with a locale-pack-driven decision. The CV language shall default to the posting language, overridable by user preference.
**Acceptance Criteria:**
- No surface hardcodes English as the only output language for generated content
- CV language is determined by the active locale pack / user preference, not a constant
- Cover-letter language continues to match the posting language (existing NFR-0006 behavior preserved)
- When the user sets a UI/content language, generation honors it
- If generation quality for the chosen language is low-resource (see REQ-7011), the caveat from REQ-7011 is applied

### REQ-7002: Externalized Static Strings
**Priority:** Must
**Description:** The system shall externalize all static user-facing strings (CLI/command output, prompts shown to users, dashboard UI chrome, error/validation messages) out of code into translation resources, with English as the source of truth.
**Acceptance Criteria:**
- No user-facing static string is hardcoded in dashboard components or command output paths
- English resource files under `i18n/ui/en/` are the canonical source
- Strings are referenced by stable semantic keys, not by English text
- UI chrome is never generated by the LLM at runtime (determinism, testability, cost)

### REQ-7003: Translation Resource Format
**Priority:** Must
**Description:** The system shall use a single documented string format and key convention: ICU MessageFormat in JSON, organized by BCP-47 language code under `i18n/ui/<code>/`.
**Acceptance Criteria:**
- Messages use ICU MessageFormat (plural, select/gender, number/date formatting)
- Plural rules cover all CLDR categories (`zero/one/two/few/many/other`) — required for Arabic, Slavic (`ru`/`uk`/`pl`), and Indic languages
- Placeholders are named (`{company}`), never positional (`%s`)
- Files are valid JSON, diff-friendly, one language directory per BCP-47 code
- The format choice and key-naming rules are documented for contributors

### REQ-7004: Language Roadmap & Tiers
**Priority:** Must
**Description:** The system shall ship a published, criteria-based language roadmap (Tier 1 launch set, Tier 2 expansion) plus a path for any community to add its language. The canonical registry lives in `i18n/_meta/languages.json`.
**Acceptance Criteria:**
- Tier 1 (launch): `en`, `es`, `zh-Hans`, `hi`, `ar`, `pt-BR`, `fr`, `de`, `ja`, `ru`, `id`, `bn`
- Tier 2 (expansion): `ur`, `fa`, `he`, `tr`, `vi`, `ko`, `it`, `pl`, `nl`, `uk`, `th`, `ms`, `ta`, `te`, `mr`, `fil`, `sw`, `ro`, `pt-PT`, `zh-Hant`
- Each registry entry records: code, English name, endonym, text direction (ltr/rtl), script, tier, and status (released/beta)
- Adding a language is a drop-in under `i18n/` + a registry entry — no core code change
- Selection rationale (reach × labor-market × mobility × growth) is documented

### REQ-7005: Translation Contribution Workflow
**Priority:** Should
**Description:** The system shall provide a translator-facing contribution workflow using Weblate, with machine-translation seeding followed by human post-editing (MTPE), reviewed like code, with contributor credit.
**Acceptance Criteria:**
- A translator CONTRIBUTING guide explains how to add/update a language without touching code
- Weblate is configured with a Git round-trip against the `i18n/` tree
- New languages are MT-seeded, then human-reviewed before being marked "released"
- Contributors are credited (git history + platform per-translator stats)
- The deep internal engineering spec is explicitly out of translation scope

### REQ-7006: Translation Parity / Staleness Gate
**Priority:** Must
**Description:** The system shall detect when a translation has fallen behind the English source and surface the gap, since no off-the-shelf tool reliably does this.
**Acceptance Criteria:**
- Each translated file records the English source commit-hash it was translated from (front-matter)
- `i18n/_meta/source-manifest.json` records the current hash per English source file
- A CI check reports, per language, how many strings/files are behind ("N behind")
- A language below the NFR-0020 completeness threshold is marked **beta**, not **released**
- The check runs in CI and is visible to maintainers and translators

### REQ-7007: README & Docs-Site Translation
**Priority:** Should
**Description:** The system shall provide localized README variants and a localized docs-site, without cluttering the repository root.
**Acceptance Criteria:**
- Root `README.md` is English (canonical) and carries a language-switcher table at the top
- Localized READMEs live under `i18n/readme/README.<code>.md` — never in the repo root
- The docs-site supports a language switcher and per-language content in its framework-native i18n layout under `docs-site/`
- The same Tier-1 language set is targeted for README + docs-site first, expanding to Tier 2
- Each localized README records the source-hash it was translated from (REQ-7006)

### REQ-7008: Dashboard UI Localization + RTL
**Priority:** Should
**Description:** The dashboard shall render in the user's chosen language with full right-to-left support and correct non-Latin typography.
**Acceptance Criteria:**
- Dashboard chrome (nav, labels, buttons, table headers, chart legends) is fully localized via `i18n/ui/`
- RTL languages (`ar`, `he`, `fa`, `ur`) render with bidi-aware layout, mirrored UI, and logical CSS properties
- CJK languages render with correct line-breaking and full-width punctuation; Indic languages use shaping-capable fonts
- Language is user-selectable; selection persists in dashboard-local settings
- Realized in tandem with NFR-0019

### REQ-7009: Locale Packs (Pluggable Region Conventions)
**Priority:** Must
**Description:** The system shall define a documented, pluggable locale-pack interface for region conventions, decoupled from UI language and from the core, in the spirit of ADR-0004 portal adapters.
**Acceptance Criteria:**
- Locale packs live under `locale-packs/<code>/` as data, against a published `_schema.json`
- A pack parameterizes at least: document term (CV/résumé/Lebenslauf/rirekisho/currículo), page-count expectation, photo norm (forbidden/optional/expected/required-with-spec), included personal fields (DOB, marital status, nationality, national ID), name ordering, date/number/currency format (CLDR), address format, and privacy/legal clauses
- No country-specific convention is hardcoded in core or templates (realizes CLAUDE.md "no hardcoded locales")
- A missing locale pack degrades gracefully to a neutral default (no photo, ISO dates, A4) per ARCH-0005
- Community can contribute a new region pack without core changes

### REQ-7010: Locale-Aware Document Generation
**Priority:** Should
**Description:** CV and cover-letter generation shall consume the active locale pack so output respects the target market's conventions.
**Acceptance Criteria:**
- Photo inclusion follows the pack (e.g., omitted for US/UK; included for DE; per spec for JP rirekisho)
- Personal fields follow the pack (e.g., CPF + marital status common in BR; DOB/nationality common in IN; omitted in US/UK)
- Page size follows the pack (A4 vs Letter); page-count expectation is surfaced to the user
- EU locale packs include a GDPR consent line where appropriate
- The drafter never fabricates locale-required fields the candidate has not provided (ARCH-0007) — it prompts the user instead

### REQ-7011: Per-Language Generation-Quality Signal
**Priority:** Should
**Description:** Because LLM generation/translation quality degrades for low-resource languages, the system shall flag lower-confidence languages and lean on human review rather than assuming uniform quality.
**Acceptance Criteria:**
- The language registry marks low-resource languages (e.g., `bn`, `ta`, `te`, `mr`, `sw`, `ur`) with a quality caveat
- For flagged languages, generated content carries a visible "please review carefully" note
- The system never silently switches to a different/related language; if it cannot produce the requested language reliably, it says so honestly (ARCH-0007)
- Existing human-in-the-loop review (ARCH-0006) is the backstop for flagged languages

## New Glossary Terms

To be added to [glossary.md](../glossary.md):
- **Localization Tree (`i18n/`)** — the single repository directory holding all static translation artifacts, keeping the root uncluttered.
- **Locale Pack** — a pluggable data file of a region's conventions (CV norms, photo, page size, formats, legal clauses), decoupled from UI language.
- **Language Registry** — `i18n/_meta/languages.json`, the canonical list of supported languages with tier, direction, script, and status.
- **Translation Staleness** — the condition where a translated file is behind its English source, detected by the source-hash parity check (REQ-7006).
- **MTPE (Machine-Translation Post-Editing)** — the seeding model: machine-translate, then human-review before release.
