# ADR-0007: Internationalization & Localization Architecture

> **Status:** Accepted
> **Date:** 2026-06-18
> **Decision makers:** Architecture team + Localization Architect
> **Supersedes:** Tightens NFR-0006 (multi-language support) into a full architecture; removes the hardcoded "CV always English" rule.
> **Related:** [Proposal 0001](../proposals/0001-competitive-additions.md), ADR-0004 (pluggable adapters — same plugin philosophy), ADR-0003 (LaTeX generation — script/font impact).

---

## Context

CareerForge's prime directive is to serve job seekers **anywhere in the world, regardless of country or language** (CLAUDE.md). Today the product is English-only at every static surface (README, docs-site, CLI output, dashboard UI), and a hardcoded rule forces every CV into English. That contradicts the directive and caps the addressable audience at English readers.

The owner's directive: make the entire product usable, readable, and translatable in as many world languages as possible — *each added language is real value to that language community* — while keeping the repository organized (no thousands of files in the root) and the engineering effort sustainable as languages scale.

A naïve approach (sprinkle `README.es.md`, `README.zh.md`, … in the root; hardcode per-country logic; let an LLM render UI chrome at runtime) would clutter the repo, drift out of sync, and be untestable. We need a deliberate architecture.

### Two distinct problems, often conflated

1. **Language** — the *words* a surface is written in (English → Spanish). Solved by translating externalized strings.
2. **Locale / region** — the *conventions* of a market (does a CV include a photo? A4 or Letter? GDPR clause? surname-first?). Solved by region config, **decoupled from language** (a Spanish speaker may be applying in Germany).

Treating these as one axis is the most common i18n mistake and the source of hardcoded-locale bugs. We separate them.

### The AI-native head start

Because user-facing **content** (CV bullets, cover letters, evaluation rationales, interview Q&A) is generated by the model at runtime, CareerForge is **already multilingual at the content layer** — the heaviest, most-churning translation surface largely disappears. What remains is the **static chrome** (UI labels, CLI help, error messages, docs), which must be deterministic and testable, so it is translated traditionally — *not* generated at runtime (runtime generation of chrome is non-deterministic, untestable, and adds latency/cost per render).

Caveat: LLM generation quality degrades for **low-resource languages** (measured gaps up to ~24%; risk of mistranslation, hallucination, or silently switching to a related language). The architecture must flag this rather than assume uniform quality.

## Decision

Adopt a four-part internationalization & localization architecture:

1. **Canonical-English source, translated user surfaces.** English is the single source of truth in version control. We **fully translate every user-facing surface** (README/quick-start, docs-site guides, CLI/command output, dashboard UI, glossary, error messages) and **keep the deep internal engineering spec English-canonical** (`docs/requirements`, `docs/architecture`, `docs/testing` — 130+ contributor-facing, fast-churning docs). This mirrors React, Vue, MDN, TypeScript, Kubernetes, and Docusaurus.
2. **One localization tree — `i18n/` — never the repo root.** All translation artifacts live under a single top-level `i18n/` directory (plus the docs-site's framework-native i18n folder). The root keeps exactly one `README.md` (English, canonical) with a language-switcher table linking into `i18n/readme/`.
3. **Locale packs are pluggable, decoupled from language.** Region conventions live in `locale-packs/<code>/` as data, consumed by document generation — the same plugin philosophy as ADR-0004 portal adapters.
4. **Weblate + ICU MessageFormat + a custom staleness gate.** Translations are managed in Weblate (FOSS, self-hostable, Git round-trip), stored as ICU MessageFormat JSON, seeded by machine translation + human post-edit (MTPE), and protected by a CI parity check that detects when a translation falls behind the English source.

### Repository organization (the "no root clutter" design)

```
repo-root/
├── README.md                       # English, canonical — with a language-switcher table at the top
├── i18n/                           # ← ALL static localization lives here, one tree
│   ├── _meta/
│   │   ├── languages.json          # tier registry: code, name, endonym, dir (ltr/rtl), tier, status, script
│   │   └── source-manifest.json    # English source file → current commit hash (staleness baseline)
│   ├── readme/
│   │   ├── README.es.md            # one localized README per language (NOT in root)
│   │   ├── README.zh-Hans.md
│   │   └── …                       # front-matter records source-hash it was translated from
│   ├── ui/                         # static UI/CLI strings, ICU MessageFormat JSON
│   │   ├── en/  { cli.json, dashboard.json, errors.json, common.json }   # source of truth
│   │   ├── es/  { … }
│   │   └── …                       # one dir per BCP-47 code
│   └── glossary/
│       └── <code>/glossary.md      # translated user-facing glossary
├── locale-packs/                   # ← REGION conventions, decoupled from language
│   ├── _schema.json                # the locale-pack contract
│   ├── us.json   de.json   jp.json   br.json   in.json   eu.json   …
└── docs-site/
    └── <framework i18n dir>/<code>/…   # docs-site translations in its framework's native layout
```

Root-directory impact: **+1 line in `README.md`** (the switcher table) and **two new top-level dirs** (`i18n/`, `locale-packs/`). Everything else nests. Adding a language adds files *inside* `i18n/`, never in the root.

### Language roadmap (maximize reach, scale sustainably)

Selection blends **internet-user reach × labor-market size/openness × cross-border mobility × growth** (Ethnologue 2025; internet-users-by-language; web-content share; growth signals). Codes are BCP-47; the registry lives in `i18n/_meta/languages.json`.

**Tier 1 — Launch (12):** `en` English · `es` Spanish · `zh-Hans` Chinese (Simplified) · `hi` Hindi · `ar` Arabic 🔁 · `pt-BR` Portuguese (Brazil) · `fr` French · `de` German · `ja` Japanese · `ru` Russian · `id` Indonesian · `bn` Bengali. *Covers the large majority of the world's internet users; adds three high-mobility/high-growth labor markets (Hindi, Bengali, Indonesian) that content-share data under-counts.*

**Tier 2 — Expansion (18):** `ur` Urdu 🔁 · `fa` Persian 🔁 · `he` Hebrew 🔁 · `tr` Turkish · `vi` Vietnamese · `ko` Korean · `it` Italian · `pl` Polish · `nl` Dutch · `uk` Ukrainian · `th` Thai · `ms` Malay · `ta` Tamil · `te` Telugu · `mr` Marathi · `fil` Filipino · `sw` Swahili · `ro` Romanian · `pt-PT` European Portuguese · `zh-Hant` Chinese (Traditional).

**North star:** *any* language, via the contribution workflow. Tiers are the seeded order, not a ceiling.

**Script / directionality engineering flags** (each a discrete work item; recorded per-language in the registry):
- 🔁 **RTL** (`ar`, `he`, `fa`, `ur`): bidi-aware layout, mirrored UI, logical CSS properties, RTL-safe LaTeX. Biggest single layout item → NFR-0019.
- **CJK** (`zh-Hans`/`zh-Hant`, `ja`, `ko`): no word-spacing, line-break rules, full-width punctuation, glyph bundling; **Simplified ≠ Traditional** (separate locales).
- **Indic shaping** (`hi`, `bn`, `ta`, `te`, `mr`): shaping-capable fonts in **both** the dashboard and the `lualatex`/`xelatex` pipeline (Noto family + `polyglossia`/`babel`).
- **Cyrillic** (`ru`, `uk`): font coverage only, low risk.

### Tooling & libraries (productivity)

| Concern | Choice | Why |
|---------|--------|-----|
| Translation management | **Weblate** (FOSS, self-host or libre hosting) | Git round-trip, no vendor lock-in, fits local-first/open ethos; per-translator credit. Crowdin free-OSS tier is the hosted alternative. |
| String format | **ICU MessageFormat** in JSON | Correct plurals/gender across CLDR categories — Arabic, Slavic, and Indic have 3–6 plural forms gettext PO can't express. Named placeholders (`{name}`), not positional. |
| Dashboard runtime (Next.js) | **next-intl** (ICU-compatible) | Native App-Router support, ICU messages, RTL-friendly. |
| Shared/CLI runtime | **FormatJS / i18next + ICU** | ICU message evaluation outside React. |
| Seeding | **MT pre-translation → human post-edit (MTPE)** | Seed many languages cheaply; humans gate before "released." |
| LaTeX multilingual typesetting | `polyglossia` (xelatex) / `babel`, Noto fonts | Bundled shaping-capable fonts for non-Latin/RTL CVs. |

### Translation staleness (the gap no tool fills)

No off-the-shelf tool reliably answers "this translation is N commits behind English." We specify a **custom CI parity check**:
- Each translated file carries front-matter `source-hash` (the English-source commit it was translated from); `i18n/_meta/source-manifest.json` records the current hash per English source file.
- CI diffs current vs recorded hash and reports per-language "N strings / files behind," mirroring the spirit of the existing `docs-site/` parity test. A translation below the NFR-0020 completeness threshold is marked **beta**, not **released**.

## Rationale

| Factor | Root-clutter / hardcoded | **This ADR (i18n tree + locale packs)** | Runtime-LLM chrome |
|--------|--------------------------|------------------------------------------|--------------------|
| Repo tidiness | ❌ N×files in root | ✅ One `i18n/` tree | ✅ none |
| Determinism / testability | ✅ | ✅ ICU + parity CI | ❌ non-deterministic |
| Plural/RTL correctness | ⚠️ ad hoc | ✅ ICU + NFR-0019 | ⚠️ model-dependent |
| Locale ≠ language | ❌ conflated | ✅ separated (locale packs) | ❌ conflated |
| Cost/latency per render | ✅ | ✅ load once | ❌ tokens per render |
| Community can add a language | ⚠️ messy | ✅ drop-in under `i18n/` + Weblate | N/A |
| Low-resource quality | — | ✅ flagged (REQ-7011) | ❌ silent failures |

**Decisive factors:** (1) the AI-native content layer removes the heaviest translation burden, so investing in clean static-chrome infrastructure is high-leverage; (2) separating locale from language is what actually delivers "country-agnostic"; (3) a single `i18n/` tree keeps the repo navigable as languages scale into the dozens.

---

## Consequences

- **Positive:** Reaches the large majority of the world's internet users at launch; repo stays tidy; locale packs make "country-agnostic" real; ICU guarantees plural/gender/RTL correctness; Weblate lets communities own their language; English contributors are unblocked (internals stay English).
- **Negative:** New CI surface (parity check) to build and maintain; RTL/CJK/Indic rendering is real engineering (UI + LaTeX fonts); MTPE needs human reviewers per language; low-resource languages need quality caveats.
- **Mitigations:** Tiered rollout (release Tier 1, beta the rest); completeness threshold (NFR-0020) gates "released"; per-language quality flag (REQ-7011) + existing human-in-the-loop (ARCH-0006) catch low-resource generation errors.
- **Fallback / graceful degradation (ARCH-0005):** A missing or incomplete translation falls back to English string-by-string (never a blank or a crash); a missing locale pack falls back to a neutral, convention-light default (no photo, ISO dates, A4).
