チャートは全体像を把握するためのものですが、テーブルこそが実際に作業する場所です。すべての
応募が1行として、新しい順に並びます。求職活動の日々の営み — *「採用担当者から電話が来た、Interview
に移そう」* — はここで行われます。

<ApplicationsTableDemo />

## 目的のものを見つける

すべての列は並べ替え可能です。フィルタはステータス、業種、職種タイプ、または
チャネルでテーブルを絞り込み、リアルタイムで更新されます。テーブルの上にある[パイプライン
ストリップ](/ja/docs/dashboard/charts#pipeline-strip)はステータスフィルタも兼ねています —
バケットをクリックすると、テーブルがそれに追従します。行をクリックすると、完全なレコードを表示する
ドロワーが開きます。すべてのフィールド、メモ、そして送信された正確な
CVとカバーレターのPDFへのリンクが含まれます。

## 編集できるのは — ちょうど2つだけ

編集できるのは **status** と **notes** のみです（*last updated* の日付は
自動で記録されます）。それ以外のすべて — 日付、会社、職種、フィット スコア、ファイル パス — は
`/apply` が書き込んだ事実の記録であり、ダッシュボードはこれらを読み取り専用として扱います。

## ステータスの状態遷移マシン

ステータスは自由形式のテキストではなく、7つの固定されたステージであり、ダッシュボードは
理にかなった遷移のみを提示します。試してみてください — ステータスを選んでみましょう:

<StatusFlowDemo />

すべてを示すと:

| 遷移元 | 遷移可能先 |
|------|------------|
| Draft | Sent, Withdrawn |
| Sent | Interview, Rejected, Withdrawn, Closed |
| Interview | Offer, Rejected, Withdrawn, Closed |
| Offer / Rejected / Withdrawn / Closed | — 最終 |

なぜこれほど厳格なのか? それは、ステータスが[面接率
KPI](/ja/docs/dashboard/kpis)と[ステータス チャート](/ja/docs/dashboard/charts)に反映されるからです —
そして、それらの数値は「Sent」が常に同じ意味を持つときにのみ読む価値があります。
不正な遷移は書き込まれる前に拒否されるため、誤クリックが密かに統計を破壊することはありません。

知っておく価値のある2つの詳細:

- **最終 = 非表示ではない。** Rejected、Withdrawn、Closed の行は薄く表示されたまま
  テーブルに残ります。あなたの履歴はデータの一部です — そして
  Rejected の山が少しずつ Interview の山へと変わっていくのを見ることは、それ自体が一種の前進です。
- **最終ステータスを間違えた?** ダッシュボードは行を最終状態から移動させません。
  あなたのトラッカーは[あなたが所有するプレーンなファイル](/ja/docs/your-data)なので、本当の
  間違いはどんな表計算アプリでもいつでも修正できます。

## 手動で行を追加する

CareerForge の外で応募した場合 — 紹介、飛び込み、メールなど? 
**+ New** ボタンで手動で行を追加でき、統計を完全に保てます。日付、
会社、職種は必須です。CVとカバーレターのフィールドは単に空のままになります。

**次へ:** [あなたのデータ](/ja/docs/your-data) — これらすべての背後にある1つのファイル。
